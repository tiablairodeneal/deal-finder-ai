# Project Context

## Purpose

Deal Finder AI is a Notion-first foundation for finding, filtering, scoring, and organizing business acquisition opportunities. It collects selected public marketplace listings, applies the acquisition criteria in `acquisition_criteria.json`, deduplicates listings, enriches each retained listing with a buy-box score and industry assessment, and saves qualified deals to the existing Notion deals database.

Notion is the durable CRM and single source of truth for deal records. This repository is the source of truth for code, criteria, scoring rules, automation instructions, and project documentation.

## Current Architecture

The core workflow is:

1. Load criteria from `acquisition_criteria.json` through `deal_finder_ai/criteria.py`.
2. Collect live public marketplace listings through `deal_finder_ai/collectors/marketplaces.py` and `deal_finder_ai/collectors/live.py`.
3. Exclude deal-breaker listings before enrichment through `deal_finder_ai/scoring.py` and `deal_finder_ai/pipeline.py`.
4. Deduplicate retained listings with `deal_finder_ai/duplicates.py`, preferring normalized listing URLs and falling back to source/title/location/price fingerprints.
5. Score buy-box fit with `deal_finder_ai/scoring.py`.
6. Classify and score industry quality with `deal_finder_ai/industry_assessment.py`.
7. Generate a short executive summary with `deal_finder_ai/summaries.py`.
8. Save qualified deals to Notion through either the connected Codex Notion connector in the scheduled heartbeat path or the local Python Notion API integration in manual/API paths.

The main local demo entry point is `run_demo.py`. The production-style CLI entry point is `python -m deal_finder_ai.production_run`.

## Production vs Provisional

Production components:

- Acquisition criteria file.
- Listing data model.
- Duplicate detection.
- Deal-breaker exclusions.
- Buy-box scoring.
- Sub-industry classification constrained to approved labels.
- Industry assessment methodology.
- Notion property mapping.
- Codex heartbeat production schedule.
- Unit tests.

Provisional or operationally limited components:

- Live scraping depends on each marketplace continuing to expose safe public pages.
- Some sources are checked but may return zero listings when static public HTML does not expose details.
- BizBuySell and BizQuest are skipped when blocked or when robots.txt cannot be safely checked.
- Axial is login-gated and is not scraped.
- The live industry research provider uses lightweight public-source fetching and heuristic scoring, not a paid research dataset.
- Static/sample data exists for tests and demos only.
- GitHub Actions is manual-only and is not the production daily runner.

## Operating Constraints

- Use Python 3.11 or newer.
- Do not add third-party dependencies unless they are necessary and documented.
- Do not bypass marketplace protections.
- Do not guess missing financial information.
- Do not commit `.env` files, tokens, caches, run artifacts, private seller information, or generated bytecode.
- The scheduled production heartbeat requires the user’s Mac, Codex app/workspace, internet access, and connected Notion OAuth to be available.

## GitHub, Codex, and Notion

GitHub stores the durable project code and documentation. Codex runs development tasks and the current scheduled heartbeat from the user’s local Mac workspace. Notion stores the actual deals database and is the CRM.

The daily production runner is currently a Codex thread heartbeat. It uses the existing local repository path and the connected Notion connector/OAuth for Notion writes. It does not require standalone `NOTION_TOKEN` or `NOTION_DEALS_DATABASE_ID` credentials for that heartbeat path.

The repository still includes a local Python Notion API integration in `deal_finder_ai/notion_sync.py`. That path is used by `run_demo.py --sync-notion`, `production_run` when not dry-run, and manual GitHub Actions production-mode tests. It requires `NOTION_TOKEN` and `NOTION_DEALS_DATABASE_ID` supplied outside source code.

## Known Documentation/Production Mismatches

- The production heartbeat uses the connected Codex Notion connector/OAuth, but the checked-in Python production CLI uses the Notion API token path when `--dry-run` is not set.
- `README.md` documents both paths because both exist. Treat the heartbeat as the current scheduled production path.
- Notion `Industry` is a multi-select with some historical broad options. The current classifier writes approved labels to `Sub-industry`, but existing older Notion rows may still contain earlier sub-industry text until refreshed or backfilled.
